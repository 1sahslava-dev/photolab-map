# PHOTO LAB by VF — Интерактивная историческая карта фотографии
### Техническое задание для Claude Code

---

## 1. Что строим

Интерактивный веб-сайт: карта мира + временная шкала развития фотографии (от камеры-обскуры до наших дней), построенные на исследовательской базе фактов PHOTO LAB by VF. Проект **растущий**: сейчас в базе 58 узлов (PL-001–PL-006), дальше будет добавляться PL-007 и т.д. Сайт должен переживать обновления данных без переписывания кода.

**Не путать с курсом**: это отдельный визуальный инструмент (для сайта/соцсетей/лендинга), а не сами карточки уроков — карточки уроков (PDF, 7 штук на тему) генерируются отдельным пайплайном и сюда не относятся.

---

## 2. Источник данных — единственный источник правды

Файл: `PHOTO_LAB_History_Map_Master_v0_3.xlsx` (или более новая версия — всегда уточнять у автора актуальную версию перед началом работы).

Все данные сайта должны генерироваться из этого файла, **не вручную вбиваться в код**. Xlsx — рабочий, "сырой", формула-driven файл; для сайта нужен конвертер xlsx → JSON (разовый скрипт, который потом просто перезапускается при каждом обновлении файла).

### Листы файла

| Лист | Назначение |
|---|---|
| `00_README` | Правила и соглашения по схеме (см. ниже) |
| `01_Master_Facts` | **Главная таблица.** Один узел = один исторический факт |
| `02_Branches` | Справочник 13 канонических веток: описание, цвет, для каких уроков |
| `03_Lessons_Map` | Привязка диапазонов узлов к урокам PL-00X (нарративная, не строгая — см. правило ниже) |
| `04_Sources` | Справочник источников (SRC-XXX): название, URL, для чего используется |
| `05_Dashboard` | Живые формулы-агрегаты (суммы по веткам/типам/урокам) — полезно для сверки, не для сайта напрямую |
| `06_Lists` | Списки для выпадающих меню (валидация) |

### Схема `01_Master_Facts` (26 колонок)

`ID` (PL-HIST-0001…), `Период/дата`, `Точность даты` (точная дата / год / диапазон / примерно / спорно-уточнить), `Страна`, `Город/регион`, `Персона/группа`, `Событие`, **`Ветка развития`** (одно из 13 значений `02_Branches`), `Технология/процесс`, `Что изменилось`, `Предыдущая связь`, `Следующая связь`, `Статус ветки`, `Важность 1–5`, **`Привязка к уроку`** (PL-001…PL-00X / будущие PL / межурочный узел), `Источник ID` (SRC-XXX), `Название источника`, `URL источника`, `Уверенность 1–5`, `Комментарий для лекции`, `Визуальная идея для карты`, `Теги`, `Проверить позже`, **`Координаты (lat, lon)`**, **`Подветка (детали)`**, **`Тип узла`** (11 значений: технология/процесс, фундаментальная предпосылка, инструмент/устройство, публикация/объявление, терминология/язык, связь/переход, тиражирование/публикация, распространение/коммерциализация, научное применение/пример, применение/жанр, ключевой исторический узел).

### Обязательные правила (записаны в `00_README`, соблюдать буквально)

1. **Ветка развития** = только одно из 13 значений `02_Branches`. Не создавать новые ветки в коде — если понадобится 14-я ветка, это решение принимает автор в самом xlsx, сайт лишь отражает текущий список.
2. **Подветка** — уточнение внутри ветки, для тултипов/подписей, не для основной цветовой кодировки.
3. **Тип узла** — определяет иконку/форму маркера на карте (роль факта: предпосылка vs технология vs публикация и т.д.), независимо от ветки (влияющей на цвет).
4. **Единственный источник истины для фильтра «по уроку» — колонка `Привязка к уроку`.** Диапазоны в `03_Lessons_Map` (например «PL-HIST-0001…0008, 0013, 0028») — нарративные, для контекста темы, узлы могут в них пересекаться между уроками. **Не использовать эти диапазоны как множество для фильтрации** — только как подпись «эта тема опирается на такие-то узлы плюс контекст».
5. **Координаты — на уровне города**, не точный адрес. У части узлов (распределённые процессы вроде «Европа/Америка», без единой точки) координаты **намеренно пустые** — для них нужен отдельный визуальный режим на карте (см. п.4 ниже), а не пропуск узла и не выдуманная точка.
6. Никогда не восстанавливать/не придумывать данные, которых нет в файле (пустые координаты, пустые источники и т.п.) — пустое поле обрабатывается как отдельное состояние UI, не как ошибка данных для исправления кодом.

## 3. Известные особенности данных, под которые дизайнить заранее

- 2 узла без координат (распределённые процессы) — нужен режим "не точка на карте", например отдельная лента/список "глобальные процессы" рядом с картой, либо иконка вне геопривязки.
- 3 узла с пограничной классификацией ветки (жанровые/прикладные факты, силой отнесённые к ближайшей технологической ветке) — помечены в `Проверить позже`; на сайте ведут себя как обычные узлы этой ветки, если автор позже заведёт для них отдельную ветку — они просто перекрасятся при следующей выгрузке, код менять не придётся.
- Координаты не подходят для сильного зума в масштабе одного города (несколько узлов Лондона/Парижа лягут в одну точку) — на сильном зуме нужна кластеризация маркеров (marker clustering), а не рассчитывать на уникальность координат.
- Данные будут расти построчно (PL-007, PL-008…) — не хардкодить количество узлов, диапазоны дат, список ID.

## 4. Функциональные требования к сайту

1. **Карта** — точки по координатам, цвет = ветка развития (13 цветов из `02_Branches`), форма/иконка маркера = тип узла. Клик/тап по точке → карточка с деталями узла (событие, персона, дата, комментарий для лекции, источник + ссылка, уверенность).
2. **Временная шкала** под/рядом с картой — те же узлы, отсортированные по дате, с учётом `Точность даты` (для "диапазон"/"примерно" — визуально показывать как интервал, не точку).
3. **Фильтры**: по уроку (`Привязка к уроку`, правило см. п.2.4), по ветке развития, по типу узла, по стране, по минимальной уверенности (1–5). Фильтры комбинируются.
4. **Список "глобальных" узлов без координат** — отдельный блок, не встроенный в карту как точка.
5. **Индикатор качества данных** (опционально, но полезно для рабочего инструмента автора) — визуальная пометка узлов с непустым `Проверить позже` или низкой `Уверенность`, чтобы автор видел, что ещё требует доработки, прямо на карте/в админ-режиме.
6. Адаптивность: рабочий инструмент для автора (десктоп) в приоритете, мобильная версия — по возможности, не критично для MVP.

## 5. Визуальный стиль (Brand Book PHOTO LAB by VF)

Сайт должен визуально продолжать бренд-бук курса, не выглядеть как отдельный несвязанный продукт.

- **Navy / Header** `#020B18` — тёмные акценты, хедер, фон карты (тёмная тема карты предпочтительна).
- **Ink** `#061126` — текст, заголовки.
- **Warm Paper** `#F7F5EF` — светлый фон панелей/карточек.
- **Gold Accent** `#C9972B` — акценты, активные фильтры, выделение выбранного узла.
- **Soft Line** `#D6D1C7` — разделители, рамки.
- **Telegram Blue** `#27A7E7` — только иконка Telegram, больше нигде.
- Пропорция: 70% тёплый белый/тёмно-синий фон, 20% графитовый текст, 3–7% золото (золото не должно превращаться в декоративный шум).
- Типографика: жирный гротеск (Inter Black/ExtraBold для заголовков, Inter Regular/Medium для текста), крупные заголовки, короткие подписи, без канцелярита.
- Иконки — линейные, одинаковой толщины (1.5–2px), без смешивания разных наборов.

## 6. Технологический стек (рекомендация, не жёсткое требование)

- Данные: скрипт-конвертер `xlsx → JSON` (Python/openpyxl или Node/xlsx), запускается вручную при каждом обновлении мастер-файла. JSON — единственное, что реально читает фронтенд.
- Фронтенд: React + Vite. Карта — Leaflet (проще, бесплатно, без ключей API) или Mapbox GL (красивее, нужен токен). Таймлайн — свой компонент или D3.
- Хостинг: Vercel/Netlify (статика, бесплатный тариф достаточен на старте).
- Git с самого начала — важно для истории изменений по мере роста данных.

## 7. Порядок работы (для Claude Code)

1. Создать проект, поднять git.
2. Написать и протестировать конвертер `xlsx → data.json` по схеме из раздела 2 — на входе актуальный мастер-файл, на выходе плоский JSON-массив узлов + справочник веток/цветов.
3. Собрать MVP: карта + список узлов без координат + базовые фильтры (ветка, урок, тип).

5. Применить визуальный стиль по разделу 5.
6. Протестировать на текущих 58 узлах, не на выдуманных данных.
7. Задокументировать в README проекта: как обновлять данные (просто положить новый xlsx и перезапустить конвертер) — это рабочий процесс на годы вперёд, не разовая сборка.

---

## 8. Важно для Claude Code

- Xlsx-файл — рабочий инструмент автора, редактируется в отдельном чате с Claude (не с Claude Code). Claude Code получает файл как **вход** для конвертера, не редактирует его исторические факты и не выдумывает новые узлы/источники/координаты от себя.
- Если каких-то данных не хватает для красивой визуализации (например, нет описания цвета для новой гипотетической ветки) — не досочинять, а явно показать пробел в интерфейсе или спросить автора.
